## Approaches Considered

### Approach A: Workflow Orchestrator with LLM API (CHOSEN)

One UI around the 11-step workflow. Save brand context once. Steps 1-8 call the LLM
API directly (OpenAI or compatible). Steps 9-11 remain external. User brings their
own API key.

- Effort: S+ (1-2 weeks solo, LLM API calls add ~1 day)
- Risk: Low — user owns the API key, no billing complexity on Gigity's side
- Ships: immediately usable on day 1, faster than manual copy-paste

### Approach B: Full API Pipeline (deferred to V2)

Full LLM + SunoAI API + KlingAI API integration.

- Effort: L (1-2 months)
- Risk: High (SunoAI/KlingAI API availability, credit cost, character consistency unsolved)

### Approach C: Brand Intelligence Layer

Product profile that generates prompts for any AI tool. Tool-agnostic. Rejected:
feels like a feature, not a product — and Approach A already includes this as its core.

---

## Recommended Approach

**Approach A — Workflow Orchestrator with LLM API**

Why: The builder is the target user. Calling the LLM API directly eliminates the
copy-paste step entirely — the biggest daily friction point. User brings their own
OpenAI key, so there's no billing complexity on Gigity's side. Steps 9-11 (KlingAI,
CapCut, upload) stay external because those tools don't have simple APIs yet.

The LLM integration is also simpler than it sounds: one POST to OpenAI's chat
completions endpoint. No SDK needed beyond a plain fetch call.

## Character Architecture Decision (CEO Review 2)

**Problem identified:** The original design had `BrandProfile.characters[]` storing
pre-defined brand characters. This assumes users pre-configure characters before the
story exists. The real workflow is inverted: the LLM generates a story, the story
creates characters organically, then character image prompts are generated from those
story characters.

### Approach A (CHOSEN for V1): Remove characters from BrandProfile entirely

Characters are story-driven, not brand-level. Delete `BrandProfile.characters[]`,
delete `activeCharacter` from VideoProject, remove character picker from "New Video"
modal. Add a new external step (step 6) between character prompts and scene prompts:
user generates character images in Midjourney/DALL-E, pastes URLs back into Gigity.
Scene image step uses those URLs as context.

- Effort: S (delete code, add one new step definition, adjust workflow count to 11)
- Risk: Low
- Pros: Matches the real workflow. Simpler brand setup form. Characters feel organic.
- Cons: No cross-video character continuity in V1.

### Approach B: Keep characters in BrandProfile as optional "recurring characters"

`characters[]` stays in BrandProfile but optional. If configured, recurring characters
get injected into the system message for step 5. Character picker in "New Video" modal
becomes optional.

- Effort: S (mostly additive)
- Risk: Low
- Pros: Supports brand mascots. User can ignore it for one-off characters.
- Cons: Two character pathways is confusing UX. More complex brand setup.

### Approach C: Remove from BrandProfile, add per-video character saving after generation

No characters at brand setup time. After approving step 5, user can optionally save
those characters back to BrandProfile for reuse in future videos.

- Effort: M (new "save to brand" flow)
- Risk: Medium
- Pros: Best of both worlds — characters emerge from stories, can persist across videos.
- Cons: V2 scope. More UI complexity.

**DECISION:** Approach A for V1. Approach B is trivial to add in V2 if the builder
wants recurring brand mascots after using Gigity for real.

## Revised 11-Step Workflow (post character redesign)

```
Step 1  → LLM API  — business context + example videos → generates campaign brief
Step 2  → LLM API  — campaign brief → generates story script
Step 3  → LLM API  — story → generates SunoAI-formatted lyrics
Step 4  → LLM API  — lyrics + brand tone → generates SunoAI music prompt
Step 5  → LLM API  — story → generates character image prompts (characters emerge from story)
Step 6  → EXTERNAL — user generates character images in Midjourney/DALL-E, pastes URLs
Step 7  → LLM API  — character images + scene breakdown → generates scene image prompts
Step 8  → LLM API  — scene image prompts + story → generates KlingAI animation prompts
Step 9  → EXTERNAL — user takes KlingAI prompts to KlingAI, uploads result URL
Step 10 → EXTERNAL — user assembles clips + music + logo in CapCut
Step 11 → EXTERNAL — user uploads to TikTok, YouTube, Instagram Reels
```

LLM steps (1-5, 7-8): user clicks Generate → LLM API called → result shown → user
can retry or follow up → user approves → next step unlocks.

External steps (6, 9-11): instruction card + external tool link + asset URL input

- Approve button. No LLM call.

## UX Model: Approve Gate + Conversation Thread (CEO Review 2)

**Decisions made:**

- **Step 1 auto-generates on /projects/[id] page load.** Create Video redirect lands
  here and step 1 is already running. User sees result immediately.

- **Approve step N → step N+1 becomes active, user clicks Generate manually.**
  No auto-pipeline. User reviews each step before the next begins.

- **Retry = fresh call** (clears `conversation[]`). Follow-up = conversation thread
  (appends to `conversation[]`, sends full history to LLM).

- **Prior approved outputs injected into system message** (not stored in
  `conversation[]`). System message is rebuilt on every generate call.

- **Approval is irreversible in V1.** No un-approve.

## Architectural Decisions

These decisions were made during `/plan-ceo-review` (2 runs) and supersede the
original data model where they conflict.

| #   | Decision                                                                                 | Rationale                                                                                                                                                                              |
| --- | ---------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | `activeCharacter` stored as snapshot (not ObjectId) on VideoProject                      | Characters are embedded subdocuments — ObjectId lookup requires full BrandProfile scan. Snapshot is simpler and keeps historical consistency.                                          |
| 2   | `generatedPrompt` computed on-the-fly, not stored in DB                                  | Prevents stale prompts after brand profile edits. Computed from workflow-templates.ts + BrandProfile + activeCharacter at render time.                                                 |
| 3   | WorkflowStep: stepNumber, conversation, llmResponse, outputAssetUrl, status, completedAt | promptTemplate and generatedPrompt removed — they're derived, not stored.                                                                                                              |
| 4   | Linear step ordering enforced                                                            | API returns 400 if step N is marked done before step N-1. Sidebar shows future steps as locked.                                                                                        |
| 5   | "New Video" → modal on dashboard                                                         | Title input in modal. No separate /projects/new route.                                                                                                                                 |
| 6   | Each workflow-template step has `type` field                                             | `'llm' \| 'external_instruction'` — controls UI rendering per step.                                                                                                                    |
| 7   | URL validation: https?:// only                                                           | logoUrl, exampleVideoUrls, outputAssetUrl validated at API layer.                                                                                                                      |
| 8   | Brand profile delete blocked if VideoProjects reference it                               | Prevents orphan VideoProject documents.                                                                                                                                                |
| 9   | `userId: null` on both models                                                            | V1: always null. V2 migration: populate from auth middleware. Zero migration cost.                                                                                                     |
| 10  | ENV split: server-only vs client                                                         | `constants/env.server.ts` (MONGODB, OPENAI*API_KEY, secrets) imported with `server-only`. NEXT_PUBLIC*\* vars in separate client-safe file.                                            |
| 11  | LLM via plain fetch, not SDK                                                             | `openai` npm package is unnecessary for one endpoint. `fetch` to the chat completions URL is enough.                                                                                   |
| 12  | LLM step type: `'llm'` replaces `'text_prompt'` and `'image_prompt'`                     | All ChatGPT steps become LLM API calls. No copy-paste prompt UI. Step type enum: `'llm' \| 'external_instruction'`.                                                                    |
| 13  | WorkflowStep stores `conversation: Message[]` + `llmResponse`                            | `conversation[]` = full thread for retry/follow-up. `llmResponse` = latest assistant response, used as `{{step_N_output}}` context. System message NOT stored — recomputed at runtime. |
| 14  | `status: "generating"` added to WorkflowStep                                             | Prevents double-submit while LLM API call is in flight. Frontend disables Generate button when status = "generating".                                                                  |
| 15  | Step 1 auto-generates on /projects/[id] page load                                        | Create Video redirect lands on workflow view, which triggers step 1 via useEffect. User sees result immediately.                                                                       |
| 16  | Approve is a separate action from Generate                                               | `POST .../approve` sets status='done' and unlocks step N+1. Generate (and retry/follow-up) are separate calls. Approval is irreversible in V1.                                         |
| 17  | Follow-up message = conversation thread, not fresh prompt                                | Retry with follow-up appends to `conversation[]` and sends full history to LLM. Fresh Retry clears `conversation[]` and starts over.                                                   |
| 18  | Prior step outputs injected as system message context, not stored in conversation[]      | System message is rebuilt on each generate call: brand profile + all done steps' `llmResponse` values. Not stored in DB — prevents stale context.                                      |
| 19  | Characters removed from BrandProfile in V1                                               | Characters emerge from the story, not brand pre-configuration. `BrandProfile.characters[]` removed. No character picker in "New Video" modal. Step 6 is a new external step.           |

## Success Criteria

- Builder produces a Deewas ad using Gigity in under 1.5 hours (vs. current 4-6)
- Brand context saved across sessions — zero re-entering of product info
- LLM steps (1-5, 7-8) generate output in one click — no manual copy-paste into ChatGPT
- At least 3 videos produced before evaluating whether to add SunoAI/KlingAI integration
