## Problem Statement

Solo app developers and indie makers (like the Deewas developer) need to create
advertising videos for TikTok, YouTube Reels, and Instagram Reels regularly. The
current process requires 5+ separate AI tools (ChatGPT, SunoAI, KlingAI, Midjourney/
DALL-E, CapCut) with no shared context, no memory between sessions, and no way to
maintain brand/character consistency across a campaign. Each video is a half-day of
tab-switching, re-explaining context, and manual assembly.

## Demand Evidence

- Builder has personally done this workflow multiple times for Deewas (deployed on
  App Store and Play Store)
- Pain is real: time (hours per video), money (multiple subscriptions), quality
  (inconsistent output with no templates or saved context)
- Landscape research confirms: "tool fragmentation" is the #1 persistent complaint
  among AI video creators in 2026, with no tool cleanly owning end-to-end workflow

## Status Quo (original manual flow)

The original 10-step manual process the builder was doing before Gigity:

1. Give ChatGPT business context — re-enter every session
2. Show ChatGPT example videos as reference
3. Ask ChatGPT for a story (gets text)
4. Ask ChatGPT to turn story into lyrics for SunoAI
5. Ask ChatGPT for a SunoAI prompt → generate music in SunoAI
6. Ask ChatGPT for character image prompts → generate in Midjourney/DALL-E
7. Per scene: ask ChatGPT for action image prompt → generate in image tool
8. Send each scene image to KlingAI → generate video clip
9. Assemble clips + music + logo in CapCut
10. Upload to TikTok, YouTube, Instagram Reels

Cost: ~4-6 hours per video. Multiple paid subscriptions. Zero context retention
between videos. Characters and brand visuals drift across videos.

## Gigity Workflow (V1 with LLM integration)

Gigity replaces steps 1-7 with direct LLM API calls using the user's OpenAI API key
(or any OpenAI-compatible endpoint). Steps 8-10 remain external — user still does
these in their respective tools.

```
Step 1  → LLM API  — business context + example videos → generates campaign brief
Step 2  → LLM API  — campaign brief → generates story script
Step 3  → LLM API  — story → generates SunoAI-formatted lyrics
Step 4  → LLM API  — lyrics + brand tone → generates SunoAI music prompt
Step 5  → LLM API  — story + characters → generates character image prompts
Step 6  → LLM API  — character image prompts + scene breakdown → generates scene image prompts
Step 7  → LLM API  — scene image prompts + story → generates KlingAI animation prompts
Step 8  → EXTERNAL — user takes KlingAI prompts to KlingAI, uploads result URL
Step 9  → EXTERNAL — user assembles clips + music + logo in CapCut
Step 10 → EXTERNAL — user uploads to TikTok, YouTube, Instagram Reels
```

User provides their own OpenAI API key (or compatible key). Gigity stores the key in
env vars — never in the database. No Gigity-managed billing.

## Target User & Narrowest Wedge

**Target user:** Solo developers and indie makers with a deployed product who run
their own social media marketing with no marketing team. They know their product deeply
but spend painful hours producing ad content.

**Narrowest wedge:** A single-product, brand-aware workflow tool. You enter your
product info once. It pre-populates all prompts for every step. You follow the workflow
in sequence. All context is saved. The output is a set of prompts + assets + a progress
tracker — you still use the AI tools, but you stop re-explaining and context-switching.

Note: V1 does NOT solve character visual consistency (Midjourney/DALL-E generates
different-looking characters even with the same prompt). V1 only provides a place to
save reference image URLs — the user still manually manages consistency.

## Premises

1. The core pain is the 10-step context-switch across 5+ AI tools to produce one video
2. The current workflow takes 4-6 hours per video and costs subscriptions to multiple services

## Data Model

### BrandProfile (MongoDB collection: `brand_profiles`)

One brand profile per product. V1 treats the first (and only) profile as the active one.
V2 can add multi-brand support with an `isActive` flag.

```
{
  _id: ObjectId,
  name: string,                    // "Deewas"
  description: string,             // "Personal financial management app with AI"
  targetAudience: string,          // "Young adults managing their first salary"
  tone: string,                    // "Warm, encouraging, modern"
  platforms: string[],             // ["TikTok", "YouTube", "Instagram Reels"]
  exampleVideoUrls: string[],      // URLs of reference/example videos
  characters: Character[],         // see below
  logoUrl: string,                 // URL entered by user (no file upload in V1)
  createdAt: Date,
  updatedAt: Date
}

Character {
  name: string,                    // "Minh"
  description: string,             // "25-year-old office worker, friendly"
  referenceImageUrl: string        // URL of approved reference image
}
```

### VideoProject (MongoDB collection: `video_projects`)

```
{
  _id: ObjectId,
  brandProfileId: ObjectId,
  activeCharacter: {               // REVISED (CEO review 2026-04-10): snapshot stored at
    name: string,                  // creation time, NOT an ObjectId reference. Characters are
    description: string,           // embedded in BrandProfile — snapshot avoids stale data
    referenceImageUrl: string      // and fragile subdocument lookup.
  } | null,                        // null when brand profile has no characters configured.
  userId: ObjectId | null,         // V1: always null. V2: set to authenticated user's _id.
  title: string,                   // "Deewas - March campaign video 1"
  status: "in_progress" | "completed",
  steps: WorkflowStep[],           // exactly 10 steps, created on VideoProject init
  createdAt: Date,
  updatedAt: Date
}

WorkflowStep {
  stepNumber: number,              // 1-10
  conversation: Message[],         // for 'llm' steps: full thread for this step
                                   // grows on each Generate/retry/follow-up call
                                   // empty array for external steps
  llmResponse: string | null,      // for 'llm' steps: the LATEST assistant response
                                   // (always = last Message with role:'assistant')
                                   // used as {{step_N_output}} context for step N+1
  outputAssetUrl: string | null,   // for 'external_instruction' steps: URL user pastes
                                   // (image/video/audio hosted on Google Drive, Dropbox, etc.)
                                   // NOTE: No file upload in V1.
                                   // WARNING: AI tool URLs expire in 24-72h — re-host before saving.
  status: "pending" | "generating" | "done",
                                   // "generating" = LLM API call in flight
  completedAt: Date | null
}

Message {
  role: "user" | "assistant",
  content: string
}
// Note: the system message (brand context + prior step outputs) is NOT stored in
// conversation[] — it is recomputed on every generate call from current brand profile
// + approved llmResponse values from prior steps. Prevents stale context.

// promptTemplate and generatedPrompt are NOT stored in DB.
// generatedPrompt is computed on-the-fly in the API from workflow-templates.ts + current
// brand profile + activeCharacter snapshot. This prevents stale prompts after brand edits.
//
// For 'llm' steps: the LLM response is stored in llmResponse after the API call.
// The user can edit llmResponse before marking the step done.
// For 'external_instruction' steps: llmResponse is null; user fills outputAssetUrl.
//
// Steps must be completed in order: step N can only be marked done if step N-1 is done.
```

### Prompt Template Format

Templates use `{{variable}}` interpolation. Available variables:

- `{{brand_name}}`, `{{brand_description}}`, `{{target_audience}}`, `{{tone}}`
- `{{character_name}}`, `{{character_description}}`, `{{character_reference}}` — from VideoProject.activeCharacter
- `{{platform}}`, `{{example_videos}}`
- `{{step_1_output}}` through `{{step_7_output}}` — the approved `llmResponse` from each prior step
  (resolves to empty string if that step is not yet done, or if it's an external step)
- All variables resolve to empty string if the source field is null/undefined.

The **system message** for every LLM call is built at runtime:

```
1. Brand context (brand_name, description, target_audience, tone, platforms, example_videos)
2. Prior approved outputs: "Step 1 output: {{step_1_output}}\nStep 2 output: ..." (only done steps)
3. Active character context (if activeCharacter is set)
```

The **user message thread** (`conversation[]`) is what the user sees and can extend with follow-up messages.

Base templates are hardcoded in V1 (not user-editable). Templates live in
`/src/lib/workflow-templates.ts` as a constant array of 10 step definitions.

Each step definition includes:

```ts
{
  stepNumber: number,
  title: string,
  tool: 'Gigity' | 'SunoAI' | 'KlingAI' | 'CapCut' | 'Manual',
  // 'Gigity' = LLM API call (steps 1-7). Others = external tools user operates manually.
  type: 'llm' | 'external_instruction',
  // type controls UI rendering:
  //   llm              → "Generate" button (calls LLM API) + editable response textarea + mark done
  //   external_instruction → instruction card + external link + asset URL input field
  promptTemplate: string,          // with {{variables}}, used as the LLM system+user prompt
  expiryWarning?: string,          // shown for asset URL fields (AI tool URLs expire 24-72h)
}
```

## Screens

1. **Brand Setup** (`/brand/new`, `/brand/[id]/edit`)

- Form: name, description, target audience, tone, platforms, logo URL (text input — no upload)
- Character management: add/remove characters with name, description, reference image URL
- Example video URL list

2. **Dashboard** (`/`)

- If no brand profile exists: redirect to `/brand/new`
- If brand profile exists: show list of video projects (most recent first)
- Empty state (brand exists, no videos): prominent "Create your first video" CTA
- "New Video" button → opens modal with: title input + character picker (if >1 characters)
  → POST /api/v1/projects → redirect to /projects/[id]
- Character picker: skip (activeCharacter: null) if brand profile has 0 characters
- Status badge per project (in_progress / completed)
- "Edit Brand" link → `/brand/[id]/edit`

3. **Workflow View** (`/projects/[id]`)

- This is the primary workflow screen. It is a SINGLE PAGE (not separate routes per step).
- **On page load**: step 1 auto-triggers its LLM call immediately (status = "generating").
  No user action needed — the redirect from Create Video brings the user here and step 1 is already running.
- Left sidebar: step list 1-10 with status icons (pending / generating / done / active)
  - `done` = green check
  - `generating` = spinner
  - `active` = highlighted (current step to act on, not yet generating)
  - `pending` = locked (prior step not done)
- Main panel: active step card — two variants:

  **`llm` step** (steps 1-7) — three sub-states:
  - `generating`: spinner, "Generating..." — Generate button disabled
  - `has result`: response shown in a read-only card (not a textarea — it's the LLM's output)
    below that: a follow-up message input ("Add a message to refine...") + Retry button + Approve button
  - `done`: response shown read-only, "✓ Approved" label — no further actions

  **`external_instruction` step** (steps 8-10):
  - Shows instruction card + external tool link + asset URL input field + Approve button

- **Approve** (for llm steps): calls `POST /api/v1/projects/[id]/steps/[n]/approve`
  → sets step.status = 'done', saves llmResponse, unlocks step N+1 (sets it to 'active')
  → step N+1 does NOT auto-generate — user sees it highlighted and clicks Generate manually

- **Retry**: calls `POST /api/v1/projects/[id]/steps/[n]/generate` with empty follow-up
  → runs LLM again from scratch (clears conversation, starts fresh)
  → replaces the response card with new result

- **Add message + send**: calls `POST /api/v1/projects/[id]/steps/[n]/generate` with followUpMessage
  → appends user message to conversation[], sends full thread to LLM
  → response replaces the card (conversation grows, only latest response shown)

- Clicking a done step in sidebar shows its approved response (read-only) — no editing after approval
- No separate `/projects/[id]/step/[n]` route
- LLM errors shown inline with Retry button — never crash the page

## Approaches Considered

### Approach A: Workflow Orchestrator with LLM API (CHOSEN)

One UI around the 10-step workflow. Save brand context once. Steps 1-7 call the LLM
API directly (OpenAI or compatible). Steps 8-10 remain external. User brings their
own API key.

- Effort: S+ (1-2 weeks solo, LLM API calls add ~1 day)
- Risk: Low — user owns the API key, no billing complexity on Gigity's side
- Ships: immediately usable on day 1, faster than manual copy-paste

### Approach B: Full API Pipeline (deferred to V2)

Full LLM + SunoAI API + KlingAI API integration.

- Effort: L (1-2 months)
- Risk: High (SunoAI/KlingAI API availability, credit cost, character consistency unsolved)

### Approach C: Brand Intelligence Layer

Product profile that generates prompts for any AI tool. Tool-agnostic. Rejected:
feels like a feature, not a product — and Approach A already includes this as its core.

## Recommended Approach

**Approach A — Workflow Orchestrator with LLM API**

Why: The builder is the target user. Calling the LLM API directly eliminates the
copy-paste step entirely — the biggest daily friction point. User brings their own
OpenAI key, so there's no billing complexity on Gigity's side. Steps 8-10 (KlingAI,
CapCut, upload) stay external because those tools don't have simple APIs yet.

The LLM integration is also simpler than it sounds: one POST /api/v1/llm endpoint
that takes a prompt and streams the response. No SDK needed beyond a plain fetch call
to OpenAI's chat completions endpoint.

## Architectural Decisions (CEO Review 2026-04-10)

These decisions were made during `/plan-ceo-review` in HOLD SCOPE mode and supersede
the original data model where they conflict.

| #   | Decision                                                                              | Rationale                                                                                                                                                                              |
| --- | ------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------- | --------------------------------------------------------- |
| 1   | `activeCharacter` stored as snapshot (not ObjectId) on VideoProject                   | Characters are embedded subdocuments — ObjectId lookup requires full BrandProfile scan. Snapshot is simpler and keeps historical consistency.                                          |
| 2   | `generatedPrompt` computed on-the-fly, not stored in DB                               | Prevents stale prompts after brand profile edits. Computed from workflow-templates.ts + BrandProfile + activeCharacter at render time.                                                 |
| 3   | WorkflowStep stripped to: stepNumber, userOutput, outputAssetUrl, status, completedAt | promptTemplate and generatedPrompt removed from embedded docs — they're derived, not stored.                                                                                           |
| 4   | Linear step ordering enforced                                                         | API returns 400 if step N is marked done before step N-1. Sidebar shows future steps as locked.                                                                                        |
| 5   | "New Video" → modal on dashboard                                                      | Title input + character picker in modal. No separate /projects/new route.                                                                                                              |
| 6   | Each workflow-template step has `type` field                                          | `'text_prompt'                                                                                                                                                                         | 'image_prompt' | 'external_instruction'` — controls UI rendering per step. |
| 7   | URL validation: https?:// only                                                        | logoUrl, referenceImageUrl, exampleVideoUrls, outputAssetUrl validated at API layer.                                                                                                   |
| 8   | Brand profile delete blocked if VideoProjects reference it                            | Prevents orphan VideoProject documents.                                                                                                                                                |
| 9   | `userId: null` on both models                                                         | V1: always null. V2 migration: populate from auth middleware. Zero migration cost.                                                                                                     |
| 10  | ENV split: server-only vs client                                                      | `constants/env.server.ts` (MONGODB, OPENAI*API_KEY, secrets) imported with `server-only`. NEXT_PUBLIC*\* vars in separate client-safe file.                                            |
| 11  | LLM via plain fetch, not SDK                                                          | `openai` npm package is unnecessary for one endpoint. `fetch` to `https://api.openai.com/v1/chat/completions` (or `LLM_BASE_URL`) is enough.                                           |
| 12  | LLM step type: `'llm'` replaces `'text_prompt'` and `'image_prompt'`                  | All ChatGPT steps become LLM API calls. No copy-paste prompt UI. Step type enum: `'llm' \| 'external_instruction'`.                                                                    |
| 13  | WorkflowStep stores `conversation: Message[]` + `llmResponse`                         | `conversation[]` = full thread for retry/follow-up. `llmResponse` = latest assistant response, used as `{{step_N_output}}` context. System message NOT stored — recomputed at runtime. |
| 14  | `status: "generating"` added to WorkflowStep                                          | Prevents double-submit while LLM API call is in flight. Frontend disables Generate button when status = "generating".                                                                  |
| 15  | Step 1 auto-generates on /projects/[id] page load                                     | Create Video redirect lands on workflow view, which triggers step 1 via useEffect or SSR-initiated call. User sees result immediately without clicking anything.                       |
| 16  | Approve is a separate action from Generate                                            | `POST .../approve` sets status='done' and unlocks step N+1. Generate (and retry/follow-up) are separate calls. Approval is irreversible in V1 — no un-approve.                         |
| 17  | Follow-up message = conversation thread, not fresh prompt                             | Retry with follow-up appends to `conversation[]` and sends full history to LLM. Fresh Retry clears `conversation[]` and starts over.                                                   |
| 18  | Prior step outputs injected as system message context, not stored in conversation[]   | System message is rebuilt on each generate call: brand profile + all done steps' `llmResponse` values. Not stored in DB — prevents stale context.                                      |

## The Assignment

Before writing a single component, do one full video run in your head. Open a doc and
write down the exact LLM prompt (system + user message) for each of the 7 LLM steps —
word for word, for a hypothetical Deewas campaign. These prompts are your
`workflow-templates.ts`. Also write the instruction text for the 3 external steps (8-10).

Then test each prompt manually against the OpenAI API (curl or Playground) before
wiring it into the app. A prompt that doesn't produce good output in isolation won't
get better inside the app.

The M1 milestone starts only after you have all 10 step definitions written and
the 7 LLM prompts validated manually.
