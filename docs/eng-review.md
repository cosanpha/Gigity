## Data Model

### BrandProfile (MongoDB: `brand_profiles`)

One brand profile per product. V1 treats the first (and only) profile as the active one.

```ts
{
  _id: ObjectId,
  name: string,                    // "Deewas"
  description: string,             // "Personal financial management app with AI"
  targetAudience: string,          // "Young adults managing their first salary"
  tone: string,                    // "Warm, encouraging, modern"
  platforms: string[],             // ["TikTok", "YouTube", "Instagram Reels"]
  exampleVideoUrls: string[],      // URLs of reference/example videos
  logoUrl: string,                 // URL entered by user (no file upload in V1)
  createdAt: Date,
  updatedAt: Date
}
// NOTE: characters[] was removed. Characters emerge from the story (CEO review 2026-04-10).
```

### VideoProject (MongoDB: `video_projects`)

```ts
{
  _id: ObjectId,
  brandProfileId: ObjectId,
  userId: ObjectId | null,         // V1: always null. V2: set to authenticated user's _id.
  title: string,                   // "Deewas - March campaign video 1"
  status: "in_progress" | "completed",
  steps: WorkflowStep[],           // exactly 11 steps, created on VideoProject init
  createdAt: Date,
  updatedAt: Date
}
// NOTE: activeCharacter removed. Characters are per-video, not brand-level.
```

### WorkflowStep (embedded in VideoProject)

```ts
WorkflowStep {
  stepNumber: number,              // 1-11
  conversation: Message[],         // for 'llm' steps: full thread for this step
                                   // grows on each Generate/retry/follow-up call
                                   // empty array for external steps
  llmResponse: string | null,      // for 'llm' steps: the LATEST assistant response
                                   // (always = last Message with role:'assistant')
                                   // used as {{step_N_output}} context for step N+1
  outputAssetUrl: string | null,   // for 'external_instruction' steps: URL user pastes
                                   // WARNING: AI tool URLs expire in 24-72h — re-host before saving.
  status: "pending" | "generating" | "done",
  completedAt: Date | null
}

Message {
  role: "user" | "assistant",
  content: string
}
```

**Key invariants:**

- System message (brand context + prior step outputs) is NOT stored in `conversation[]`.
  It is recomputed on every generate call from current brand profile + done steps' `llmResponse`.
- `promptTemplate` and `generatedPrompt` are NOT stored in DB. Computed on-the-fly.
- Steps must be completed in order: step N can only be approved if step N-1 is done.
- `llmResponse` = always the last `Message` with `role:'assistant'` in `conversation[]`.

## Prompt Template Format

Templates use `{{variable}}` interpolation. Available variables:

- `{{brand_name}}`, `{{brand_description}}`, `{{target_audience}}`, `{{tone}}`
- `{{platform}}`, `{{example_videos}}`
- `{{step_1_output}}` through `{{step_8_output}}` — the approved `llmResponse` from each prior step
  (resolves to empty string if that step is not yet done, or if it's an external step)
- All variables resolve to empty string if the source field is null/undefined.

The **system message** for every LLM call is built at runtime:

```
1. Brand context (brand_name, description, target_audience, tone, platforms, example_videos)
2. Prior approved outputs: "Step 1 output: {{step_1_output}}\nStep 2 output: ..." (done steps only)
```

The **user message thread** (`conversation[]`) is what the user sees and can extend with follow-up messages.

Base templates are hardcoded in V1 (not user-editable). Live in `/src/lib/workflow-templates.ts`.

Each step definition:

```ts
{
  stepNumber: number,
  title: string,
  tool: 'Gigity' | 'SunoAI' | 'KlingAI' | 'CapCut' | 'Midjourney' | 'Manual',
  type: 'llm' | 'external_instruction',
  promptTemplate: string,          // with {{variables}}
  externalLink?: string,           // for external_instruction steps
  expiryWarning?: string,          // shown for asset URL fields (AI tool URLs expire 24-72h)
}
```

## API Routes

### `POST /api/v1/projects`

Creates a new VideoProject. Body: `{ brandProfileId, title }`. Initializes 11 WorkflowSteps.

### `POST /api/v1/projects/[id]/steps/[n]/generate`

Runs the LLM call for step N. Body:

```ts
{
  followUpMessage?: string,  // if present: append to conversation[], send full thread
  retry?: boolean            // if true: clear conversation[], start fresh
}
```

Behavior:

- First call (no followUpMessage, no retry): conversation = [user message from template]
- Follow-up: appends user message to conversation[], sends full history to LLM
- Retry: clears conversation[], sends fresh call
- Sets `status = "generating"` before API call, resets to "pending" on error, sets `llmResponse` + last conversation msg on success

Validation:

- Returns 400 if step is `external_instruction`
- Returns 400 if step N-1 not done
- Returns 409 if step already `"generating"` (prevents double-submit)
- Returns 502 on LLM API error (resets step status to "pending")
- Returns 404 if project not found

### `POST /api/v1/projects/[id]/steps/[n]/approve`

Approves step N. No body required for `llm` steps. Body for external steps: `{ outputAssetUrl }`.

Behavior:

- Sets `step.status = 'done'`
- Sets `step.completedAt = now`
- Unlocks step N+1 (does NOT auto-generate — user clicks Generate manually)
- Step 11 approve → sets `project.status = 'completed'`

Validation:

- Returns 400 if llm step has no `llmResponse` (never generated)
- Returns 400 if external step has empty `outputAssetUrl`
- Returns 400 if step N-1 not done
- Returns 200 if already approved (idempotent)

## Screens

### 1. Brand Setup (`/brand/new`, `/brand/[id]/edit`)

Form fields: name, description, target audience, tone, platforms (checkboxes), logo URL
(text input — no upload), example video URL list.

No character section (removed in CEO review).

### 2. Dashboard (`/`)

- If no brand profile: redirect to `/brand/new`
- If brand profile exists: list of video projects (most recent first)
- Empty state (brand exists, no videos): prominent "Create your first video" CTA
- "New Video" button → modal with title input → POST /api/v1/projects → redirect to /projects/[id]
- No character picker (characters removed from BrandProfile)
- Status badge per project (in_progress / completed)
- "Edit Brand" link → `/brand/[id]/edit`

### 3. Workflow View (`/projects/[id]`)

Single page (not separate routes per step).

**On page load:** step 1 auto-triggers its LLM call via `useEffect`. No user action needed.

**Left sidebar:** step list 1-11 with status icons:

- `done` = green check
- `generating` = spinner
- `active` = highlighted (current step to act on)
- `pending` = locked

**Main panel — `llm` step (steps 1-5, 7-8) — three sub-states:**

- `generating`: spinner, "Generating..." — Generate button disabled
- `has result`: response shown read-only + follow-up message input + Retry button + Approve button
- `done`: response shown read-only, "✓ Approved" label — no further actions

**Main panel — `external_instruction` step (steps 6, 9-11):**

- Instruction card + external tool link + asset URL input + Approve button

**Actions:**

- **Approve** (llm): `POST .../approve` → step done, step N+1 unlocked (not auto-generated)
- **Retry**: `POST .../generate` (retry: true) → clears conversation, fresh call
- **Follow-up + send**: `POST .../generate` (followUpMessage) → appends to thread, new response shown
- **Clicking done step in sidebar**: shows approved response read-only

LLM errors shown inline with Retry button — never crash the page.

## Env Vars

```bash
# Required
MONGODB_URI=mongodb://localhost:27017/gigity
OPENAI_API_KEY=<your_openai_api_key>

# Optional
LLM_BASE_URL=https://api.openai.com/v1    # override for Ollama, Together AI, etc.
LLM_MODEL=gpt-4.1-mini                     # default model

# NextAuth (V2 only — not needed for V1)
NEXTAUTH_SECRET=
NEXTAUTH_URL=
```

`OPENAI_API_KEY` and `MONGODB_URI` live in `src/constants/env.server.ts`, imported
with `server-only`. Never exposed to client.

## The Assignment

Before writing a single component, do one full video run in your head. Open a doc and
write down the exact LLM prompt (system + user message) for each of the 7 LLM steps —
word for word, for a hypothetical Deewas campaign. These prompts are your
`workflow-templates.ts`. Also write the instruction text for the 4 external steps (6, 9-11).

Then test each prompt manually against the OpenAI API (curl or Playground) before
wiring it into the app. A prompt that doesn't produce good output in isolation won't
get better inside the app.

M1 starts only after all 11 step definitions are written and the 7 LLM prompts
validated manually.

## Test Plan

### Affected Pages/Routes

- `/brand/new` — brand profile creation form
- `/brand/[id]/edit` — brand profile edit form
- `/` — dashboard (redirect if no brand, project list, New Video modal)
- `/projects/[id]` — workflow view (auto-starts step 1, conversation thread per step)
- `POST /api/v1/projects` — creates project
- `POST /api/v1/projects/[id]/steps/[n]/generate` — LLM call (first gen or follow-up)
- `POST /api/v1/projects/[id]/steps/[n]/approve` — approves step, unlocks step N+1

### Key Interactions to Verify

- Create brand profile → save → redirect to dashboard → brand name visible in header
- Edit brand profile → save → next project load shows updated prompts in workflow view
- Click "New Video" → modal opens → fill title → submit → redirect to /projects/[id]
- **Auto-start**: /projects/[id] page load → step 1 immediately starts generating (spinner shown, no click needed)
- Step 1 generates → response card shown with: Retry button, follow-up input, Approve button
- Click "Approve" on step 1 → step 1 turns green, step 2 becomes active (highlighted) with Generate button
- Click "Generate" on step 2 → step 2's LLM call fires → response shown
- Step 2's prompt includes step 1's approved output as context
- Add follow-up message on step 3 → "Make it shorter" → send → LLM responds with refinement
- Click "Retry" on step 4 → conversation clears → fresh LLM call → new response shown
- Click approved step in sidebar → shows its approved response (read-only, no editing)
- External step (step 6) → instruction card + Midjourney link + asset URL input + Approve
- Complete all 11 steps → project shows "Completed" badge on dashboard

### Edge Cases

- New Video modal: no character picker (characters removed from brand profile)
- Brand profile form with invalid URL in logoUrl → inline validation error
- Approve with no LLM response (step never generated) → API returns 400
- Approve external step with empty outputAssetUrl → API returns 400
- Step 11 approve → project.status flips to 'completed'
- OPENAI_API_KEY missing or invalid → inline error on auto-start, Retry button shown
- LLM API rate limit hit → error shown inline, Retry available
- Double-click Generate → second click disabled while status = "generating"
- Follow-up message is empty string → send button disabled (no empty follow-up calls)
- Try to approve step 3 before step 2 is done → API returns 400
- Refresh page mid-generate (status = "generating") → page shows "Generating..." with Retry option

### Critical Paths

- Brand profile setup (create once) → persists across sessions (refresh, data remains)
- Create Video → auto-generates step 1 → approve → step 2 active → generate → approve → ... → complete
- Step 1 system message: brand_name, brand_description, target_audience, tone, example_videos
- Step 2 system message: same brand context + "Step 1 output: [approved step 1 response]"
- Step 7 system message: brand context + steps 1-6 outputs (step 6 is external — outputAssetUrl injected)
- OPENAI_API_KEY is read from env server-side only — never exposed to client

### Unit Tests (M2) — `src/lib/interpolate.test.ts`

1. All brand variables present → correct substitution in output string
2. `undefined` var → empty string (not "undefined" literal)
3. `null` var → empty string
4. Unknown `{{unknown_var}}` in template → passes through unchanged
5. Empty template string → empty string output
6. Multiple occurrences of same variable → all replaced
7. Template with no variables → returned unchanged
8. `{{step_1_output}}` with step 1 done → injects step 1 llmResponse
9. `{{step_2_output}}` with step 2 NOT done → resolves to empty string

### Unit Tests (M2) — `src/lib/llm.test.ts`

10. `buildSystemMessage()` with no prior steps done → brand context only, no step outputs
11. `buildSystemMessage()` with steps 1-3 done → includes all 3 approved outputs in system message
12. `buildMessages()` first generate → returns `[{role:'system',...}, {role:'user',...}]`
13. `buildMessages()` follow-up → returns `[{role:'system',...}, {role:'user',...}, {role:'assistant',...}, {role:'user', content: followUp}]`
14. `buildMessages()` after retry (conversation cleared) → returns `[{role:'system',...}, {role:'user',...}]` only
15. `callLLM()` with `OPENAI_API_KEY` missing → throws ConfigError (no network call)
16. `callLLM()` with `LLM_BASE_URL` set → uses custom base URL
17. `callLLM()` with `LLM_MODEL` set → uses custom model in request body
18. Fetch failure → rejects with descriptive error, not raw fetch TypeError

### API Route Tests (M2)

**`POST /api/v1/projects/[id]/steps/[n]/generate`**

- First call (no followUpMessage): conversation = [user message]; calls LLM → assistant appended → 200
- Follow-up (followUpMessage = "make it shorter"): appends user msg → LLM called with full history → 200
- Retry (retry: true): clears conversation → fresh call → 200
- External step (step 6, 9-11): returns 400 ("external step has no LLM call")
- Step N-1 not done: returns 400 ("complete step N-1 first")
- Step already "generating": returns 409 (prevents double-submit)
- LLM API error: returns 502, step status reset to "pending" (NOT "generating")
- Project not found: returns 404

**`POST /api/v1/projects/[id]/steps/[n]/approve`**

- Step is done (has llmResponse): returns 200, step.status = "done", step N+1 unlocked
- Step has no llmResponse (never generated): returns 400
- External step with empty outputAssetUrl: returns 400
- Already approved step: returns 200 (idempotent)
- Step N+1 does NOT auto-generate on approve — verified by checking step N+1 status = "pending" after approve
- Step 11 approve → project.status flips to "completed"
